Food Journal

UI, UX Design, UX Research, Front-End Development

Overview

Food Journal is a mobile app that allows users to select multiple input methods to log their food. The purpose of the app is to aid Professor Daniel Epstein in his research project about how self-tracking affects the way people track their food intake.

Duration: January 2019-June 2019
Platform Mobile App, Google Home, Amazon Echo
Skills: User Research, Wireframes, Prototyping, Design, HTML, CSS, JavaScript
Context: Research

Problem Statement

Through Professor Epstein’s research, we want to discover which logging method people tend to use when tracking food and why they gravitate towards that method. There wasn’t a food tracking mobile app available that had all the input methods the team wanted to research on.

Research

Competitive Analysis

Figure 1. Lose It(left), Myfitnesspal(middle), Instagram(right)

I took a look at food logging apps such as Lose it and Myfitnesspal to see how users interact with the database and barcode inputs. I also used Instagram to see how images are stored on a profile to help with the image input.

How Are People Logging Over Virtual Assistants?

I first needed to understand how Google Home and Amazon Alexa work. I dedicated time reading documentation and testing different apps that connect with these devices. Voice commands have a pattern where there needs to be a start for a command, an action, and other information.

Ex: “Hey Google, play some music on Spotify”

I needed to define which actions should be allowed through voice and how many different ways can the user say the same action. We wanted to add a sign in action, logging action, and a logout action.

Testing

Usability Test And Results

Figure 3. Wireframe of original logging process

Due to the timeframe of the project, I tested the low-fidelity prototype with my team. When I presented the wireframes for the logging process, the original idea was to have the user select all of the input methods, complete each one, and then go to the review page. After discussing with my team, I realized that my original idea goes against Nielson’s Heuristics of Design which are feedback and recognition vs recall. The original design makes it hard for the user to remember which inputs were chosen especially if they picked multiple options.

We decided it would be best for the user to pick one input method, complete it, and then go back to a review page where they can view the input or add another option. This approach helps users know what action was chosen so far.

Redesigning The Prototype

Figure 4. Original hi-fidelity mockups for Food Journal

The user interface of the first prototype I made had the functionality needed to conduct the research but I decided to cross-check it with platform conventions, WCAG guidelines, and Nielsons heuristics for UI. The color palette did not pass the color contrast. Also, the Log a Meal button didn’t look like a button but instead a logo. The journal icon on the navigation bar was also confusing to a first time user as it is used less commonly. I feared that the participants wouldn’t be able to make entries without help from someone else.

In order to improve the design, I needed to make an appropriate button, change icons, and choose colors that were not high in contrast. Making an app aesthetically pleasing and understandable will attract users to keep using the app and we want our participants to not miss a day of logging.

Focus Areas

- Provide appropriate feedback throughout the logging process

- Make the logging button more obvious

Solutions

Figure 5. Login Screen

User Login

To track individual data from participants, each user will be assigned a unique id to log in. Since the participant group is small, there is no need for a password as we are assuming the participants will not be sharing information with one another.

View Past Entries

Users can view past entries through the calendar tab. Having previous entries available to users will allow them to identify any patterns when logging and help them engage in conversation with researchers after the study.

Figure 6. Calendar Screen

Figure 7. Logging Process

Different Logging Methods

Users make an entry by selecting an input method and review it on the confirmation page. The user is then able to add more inputs or confirm the post.

Each input takes one to two steps to complete and it was important to keep the methods with the same amount of effort in order to prevent any skewed results.

Virtual Assistants (Google Home and Amazon Alexa)

Users can log their food by using Google Home or Amazon Echo. By saying "Log", the user starts to make an entry. All entries are viewable through the app.

Reflection

I learned how important it is to get the perspective of developers when designing UX/UI solutions. For example, I learned that code can limit what can be done in UI designs if there is a specific framework being used. Always know what tools are being used to base the designs around that. It is also extremely important to cross-check the designs with platform documentation (Android and Apple standards) and Web Content Accessibility Guidelines (WCAG).

If I could rework the design, I would add a delete/edit entry feature on assistive devices and the mobile app. It would give users more control of their entries and undo a mistake. If more time was allotted, I would’ve conducted user research to know what users like and dislike about self-tracking apps to know how to structure the interface better.
