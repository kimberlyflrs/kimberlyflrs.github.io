Food Journal: A Research Tool

Food Journal is an application that can be run on mobile, web, Google Home, Amazon Alexa, and Apple watch. I was in charge of designing and programming the mobile app and personal assistant devices.

Duration: January 2019 - June 2019

Problem

  • There is no application available that considers all of the possible input methods.
  • Other applications will also have access to user's personal data.
  • We can't access user data from another application unless an API is used.

Goal

  • Create an application that accounts all input methods.
  • Make each input method as simple as possible to avoid bias.
  • Allow participants access to their entries for later engagement.

Intro

Food Journal is a mobile app that allows users to select multiple input methods to log their food. The purpose of the app is to aid Professor Daniel Epstein from the University of Caifornia, Irvine Informatics Department in his research project about how self-tracking affects the way people track their food intake.

My Role

I was responsible for creating an interface for the mobile app and personal assistant devices. I conducted research on food tracking apps to understand how they log entries. I tested my designs for usability, created the information architecture with my team, and coded the application for use.

1. Project Requirements

After a stakeholder meeting, we wanted to create an application where

  • Only participants have access to the application
  • Keep the logging process as simple as possible for mobile and personal assistant devices

Design Actions

  • How can we make the logging process simple?

    • Allow participants to add an entry.
    • Show how current entry before submitting.
    • Keep each input short

Competitive Analysis

I took a look at food logging apps such as Lose it and Myfitnesspal to see how users interact with the database and barcode inputs. I also used Instagram to see how images are stored on a profile to help with the image input.

How Are People Logging Over Virtual Assistants?

I needed to understand how Google Home and Amazon Alexa work. I dedicated time reading documentation and testing different apps that connect with these devices. Voice commands have a pattern where there needs to be a start for a command, an action, and other information.

Ex: “Hey Google, play some music on Spotify”

I needed to define which actions should be allowed through voice and how many different ways can the user say the same action. We wanted to add a sign in action, logging action, and a logout action.

5. Testing

Usability Test

Due to the timeframe of the project, I tested the low-fidelity prototype for the mobile app and assistant devices with my team. I tested the logging process with my team. The original plan for logging was to have the user select all of the input methods, complete each one, and then go to a review page before submitting the entry.

For the assistive devices, users would open the app and say “Log [food item(s)]”. Then would be repeated what they just logged by the device.

Heuristic Evaluation and WCAG Guidelines

I compared the prototype against Nielsons heuristics and WCAG guidelines to see if the mobile application meets their criteria.

Results

Figure 3. Wireframe of original logging process

  • Logging on Google Home and Amazon Alexa was simple.
  • Difficult to remember logging inputs. The logging method makes it difficult for users to remember which inputs were chosen especially if they picked multiple options. Goes against Recognition vs Recall Heuristic.
  • No feedback when logging. When logging, it doesn’t tell the user which step they are on and have to wait until they get to the review page. Goes against Feedback heuristic.
  • Logging button not noticeable. My team said the logging button looks more like a logo.
  • Color Palette did not pass WCAG guidelines

6. Iteration Based on Findings

Figure 4. Original hi-fidelity mockups for Food Journal

Logging Flow

After talking to my team for ideas, we decided it would be best for the user to pick one input method, complete it, and return to a review page where they can either submit or add another method.

Palette Change

The bright colors proved to be distracting so I changed it to a softer color palette. Making an app aesthetically pleasing and understandable will attract users to keep using the app and we want our participants to not miss a day of logging.

7. Design Outcome

Figure 5. Login Screen

User Login

To track individual data from participants, each user will be assigned a unique id to log in. Since the participant group is small, there is no need for a password as we are assuming the participants will not be sharing information with one another.

View Past Entries

Users can view past entries through the calendar tab. Having previous entries available to users will allow them to identify any patterns when logging and help them engage in conversation with researchers after the study.

Figure 6. Calendar Screen

Figure 7. Logging Process

Different Logging Methods

Users make an entry by selecting an input method and review it on the confirmation page. The user is then able to add more inputs or confirm the post.

Each input takes one to two steps to complete and it was important to keep the methods with the same amount of effort in order to prevent any skewed results.

Virtual Assistants (Google Home and Amazon Alexa)

Users can log their food by using Google Home or Amazon Echo. By saying "Log", the user starts to make an entry. All entries are viewable through the app.

Figure 8. Assistive Device interface

Future Iteration

If I could rework the design, I would add a delete/edit entry feature on assistive devices and the mobile app. It would give users more control of their entries and undo a mistake.

What I Learned

  • User Research from the start saves time. I believe that if I were to ask users their thoughts on self-tracking apps, it would have allowed me to structure the interface better.
  • Cross-checking designs with Web and platform guidelines will improve interface for all platforms.
  • Sitemaps and information architectures allow for better understanding on how an application will work. Thanks to these visuals, I was able to know how each screen would interact with one another and what features were possible.
